39 * The svZeroDSolver is a fast simulation tool for modeling the hemodynamics of
40 * vascular networks using zero-dimensional (0D) lumped parameter models.
41 *
42 * * <a href="https://github.com/StanfordCBCL/svZeroDPlus">Source
43 * repository</a>
44 * * <a href="https://simvascular.github.io">About SimVascular</a>
45 *

49 * \draw (1,0) node[anchor=south]{$P_{in}$}

59 * # Background
60 *
61 * Zero-dimensional (0D) models
62 * are lightweight methods to simulate bulk hemodynamic quantities in the
63 * cardiovascular system. Unlike 3D and 1D models, 0D models are purely
64 * time-dependent; they are unable to simulate spatial patterns in the
65 * hemodynamics. 0D models are analogous to electrical circuits. The flow rate
66 * simulated by 0D models represents electrical current, while the pressure
67 * represents voltage. Three primary building blocks of 0D models are resistors,
68 * capacitors, and inductors Resistance captures the viscous effects of blood
69 * flow, capacitance represents the compliance and distensibility of the vessel
70 * wall, and inductance represents the inertia of the blood flow. Different
71 * combinations of these building blocks, as well as others, can be formed to
72 * reflect the hemodynamics and physiology of different cardiovascular
73 * anatomies. These 0D models are governed by differential algebraic equations
74 * (DAEs).
75 *
76 * # Architecture
77 *
78 * ## Model
79 *
80 * The solver uses a highly modular framework to model the vascular anatomy,
81 * using individual classes to represent different 0D elements of the
82 * model. The elements are part of the MODEL namespace. Currently
83 * supported elements are:
84 *
85 * #### Vessels
86 *
87 * * MODEL::BloodVessel: RCL blood vessel with optional stenosis.
88 *
89 * #### Junctions
90 *
91 * * MODEL::Junction: Mass conservation junction element.
92 * * MODEL::ResistiveJunction: Resistive junction element.
93 * * MODEL::BloodVesselJunction: RCL junction element with optional stenoses.
94 *
95 * #### Boundary conditions
96 *
97 * * MODEL::FlowReferenceBC: Prescribed flow boundary condition.
98 * * MODEL::PressureReferenceBC: Prescribed pressure boundary condition.
99 * * MODEL::ResistanceBC: Resistance boundary condition.
100 * * MODEL::WindkesselBC: RCR Windkessel boundary condition.
101 * * MODEL::OpenLoopCoronaryBC: Open Loop coronary boundary condition.
102 * * MODEL::ClosedLoopCoronaryBC: Closed Loop coronary boundary condition.
103 * * MODEL::ClosedLoopHeartPulmonary: Heart and pulmonary circulation model.
104 *
105 * The elements are based on the parent MODEL::Block class. More information
106 * about the elements can be found on their respective pages. The elements are
107 * connected to each other via nodes (see MODEL::Node). Each node corresponds to
108 * a flow and a pressure value of the model. The MODEL::DOFHandler handles the
109 * degrees-of-freedom (DOF) of the system by assigning DOF indices to each
110 * element that determine the location of the local element contribution in the
111 * global system. The complete model is stored and managed by the MODEL::Model
112 * class.
113 *
114 * ## Algebra
115 *
116 * Everything related to solving the system of equation is contained in the
117 * ALGEBRA namespace. The ALGEBRA::Integrator
118 * handles the time integration scheme (see Integrator documentation for
119 * more information on that).
120 *
121 * ## Input/output
122 *
123 * Finally the IO namespace provides everything related to reading and writing
124 * files. The IO::ConfigReader reads the simulation config and IO::to_vessel_csv and
125 * IO::to_variable_csv are two different methods for writing the result files.
126 *
127 * # Build svZeroDSolver
128 *
129 * svZeroDSolver uses CMake to build the tool. The dependencies are installed
130 * by CMake. Currently supported are:
131 *
132 * * Ubuntu 18
133 * * Ubuntu 20
134 * * Ubuntu 22
135 * * macOS Big Sur
136 * * macOS Monterey
137 *
138 * @note If you want to use the Python interface of svZeroDPlus make
139 * sure to **activate the correct Python environment** before building the
140 * solver. Also make sure to install the Python package using the `-e` options,
141 * i.e.
142 * \code
143 * pip install -e .
144 * \endcode
145 *
146 * ## Build in debug mode
147 *
148 * \code
149 * mkdir Debug
150 * cd Debug
151 * cmake -DCMAKE_BUILD_TYPE=Debug ..
152 * cmake --build .
153 * \endcode
154 *
155 * ## Build in release mode
156 * \code
157 * mkdir Release
158 * cd Release
159 * cmake -DCMAKE_BUILD_TYPE=Release ..
160 * cmake --build .
161 * \endcode
162 *
163 * ## Build on Sherlock
164 * \code
165 * module load cmake/3.23.1 gcc/12.1.0 binutils/2.38
166 * mkdir Release
167 * cd Release
168 * cmake -DCMAKE_BUILD_TYPE=Release -DCMAKE_CXX_COMPILER=/share/software/user/open/gcc/12.1.0/bin/g++ -DCMAKE_C_COMPILER=/share/software/user/open/gcc/12.1.0/bin/gcc ..
169 * cmake --build .
170 * \endcode
171 *
172 * # Run svZeroDSolver
173 *
174 * After building svZeroDSolver, the build folder contains an executable
175 * called `svzerodsolver`. Run it with one of:
176 *
177 * \code
178 * ./svzerodsolver path/to/config.json path/to/output.csv
179 * \endcode
180 *
181 * `path/to/config.json` and `path/to/output.csv` should be replaced by the
182 * correct paths to the input and output file, respectively.
183 *
184 * ## Simulation parameters
185 *
186 * The svZeroDSolver can be configured with the following options in the
187 * `simulation_parameters` section of the input file. Parameters without a
188 * default value must be specified.
189 *
190 *
191 * Parameter key | Description | Default value
192 * --------------------------------------- | ----------------------------------------- | -----------
193 * number_of_cardiac_cycles | Number of cardiac cycles to simulate | -
194 * number_of_time_pts_per_cardiac_cycle | Number of time steps per cardiac cycle | -
195 * absolute_tolerance | Absolute tolerance for time integration | \f$10^{-8}\f$
196 * maximum_nonlinear_iterations | Maximum number of nonlinear iterations for time integration | \f$30\f$
197 * steady_initial | Toggle whether to use the steady solution as the initial condition for the simulation | true
198 * output_variable_based | Output solution based on variables (i.e. flow+pressure at nodes and internal variables) | false
199 * output_interval | The frequency of writing timesteps to the output (1 means every time step is written to output) | \f$1\f$
200 * output_mean_only | Write only the mean values over every timestep to output file | false
201 * output_derivative | Write time derivatives to output file | false
202 * output_last_cycle_only | Write only the last cardiac cycle | false
203 *
204 *
205 * # Developer Guide
206 *
207 * If you are a developer and want to contribute to svZeroDSolver, you can find
208 * more helpful information in our [Developer Guide](docs/cpp/pages/developer_guide.md).
209 *

223/**
224 *
225 * @brief Run svZeroDSolver with configuration
226 *
227 * 1. Read the input file
228 * 2. Create the 0D model
229 * 3. (Optional) Solve for steady initial condition
230 * 4. Run simulation
231 * 5. Write output to file
232 *
233 * @param json_config Path config or json encoded string with config
234 * @return Result as csv encoded string
235 */
236const std::string run(std::string& json_config) {
237 // Load model and configuration
238 DEBUG_MSG("Read configuration");
239 IO::ConfigReader<T> reader;
240 reader.load(json_config);
241
242 // Setup system
243 ALGEBRA::State<T> state = reader.initial_state;
244
245 // Create steady initial
246 if (reader.sim_steady_initial) {
247 DEBUG_MSG("Calculate steady initial condition");
248 T time_step_size_steady = reader.sim_cardiac_cycle_period / 10.0;
249 reader.model->to_steady();
250 ALGEBRA::Integrator<T> integrator_steady(*reader.model, time_step_size_steady, 0.1, reader.sim_abs_tol, reader.sim_nliter);
251 for (int i = 0; i < 31; i++) {
252 state = integrator_steady.step(state, time_step_size_steady * T(i),*reader.model);
253 }
254 reader.model->to_unsteady();
255 }
256
257 // Set-up integrator
258 DEBUG_MSG("Setup time integration");
259 ALGEBRA::Integrator<T> integrator(*reader.model, reader.sim_time_step_size,
260 0.1, reader.sim_abs_tol, reader.sim_nliter);
261
262 // Initialize loop
263 std::vector<ALGEBRA::State<T>> states;
264 std::vector<T> times;
265 if (reader.output_last_cycle_only) {
266 int num_states = reader.sim_pts_per_cycle / reader.output_interval + 1;
267 states.reserve(num_states);
268 times.reserve(num_states);
269 } else {
270 int num_states = reader.sim_num_time_steps / reader.output_interval + 1;
271 states.reserve(num_states);
272 times.reserve(num_states);
273 }
274 T time = 0.0;
275
276 // Run integrator
277 DEBUG_MSG("Run time integration");
278 int interval_counter = 0;
279 int start_last_cycle = reader.sim_num_time_steps - reader.sim_pts_per_cycle;
280 if ((reader.output_last_cycle_only == false) || (0 >= start_last_cycle)) {
281 times.push_back(time);
282 states.push_back(std::move(state));
283 }
284 for (int i = 1; i < reader.sim_num_time_steps; i++) {
285 // DEBUG_MSG("Integration time: "<< time << std::endl);
286 state = integrator.step(state, time, *reader.model);
287 interval_counter += 1;
288 time = reader.sim_time_step_size * T(i);
289 if ((interval_counter == reader.output_interval) ||
290 (reader.output_last_cycle_only && (i == start_last_cycle))) {
291 if ((reader.output_last_cycle_only == false) || (i >= start_last_cycle)) {
292 times.push_back(time);
293 states.push_back(std::move(state));
294 }
295 interval_counter = 0;
296 }
297 }
298
299 // Make times start from 0
300 if (reader.output_last_cycle_only) {
301 T start_time = times[0];
302 for (auto& time : times) {
303 time -= start_time;
304 }
305 }
306
307 // Write csv output string
308 DEBUG_MSG("Write output");
309 std::string output;
310 if (reader.output_variable_based) {
311 output = IO::to_variable_csv<T>(times, states, *reader.model,
312 reader.output_mean_only,
313 reader.output_derivative);
314 } else {
315 output =
316 IO::to_vessel_csv<T>(times, states, *reader.model,
317 reader.output_mean_only, reader.output_derivative);
318 }
319 DEBUG_MSG("Finished");
320 return output;
321}
